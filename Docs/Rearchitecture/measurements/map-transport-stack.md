# map:transport-stack

## Summary

Transport is a real variation axis and its absorption point already exists structurally â€” but only at `package` visibility. WebInspectorTransport is a pure-Foundation target (10 files, all `import Foundation`, 2,407 lines) with ZERO `public` declarations; its seam is the 2-method `package protocol TransportBackend` (sendJSONString/detach) consumed by the concrete `package actor TransportSession` (947 lines), which InspectorSession depends on directly (concrete type, not a protocol). The native stack plugs into that seam at a single composition root, `InspectorSession.attach(to: WKWebView)` in WebInspectorNativeTransport, and tests prove the seam works without the native bridge (FakeTransportBackend drives full InspectorSession connects). An external consumer today cannot build a remote/WebSocket transport for three access-control reasons, not architectural ones: TransportBackend/TransportSession/ProtocolCommand are package-only, InspectorSession and its connect(transport:) are package-only, and the only public attach path hard-codes the native factory chain. One structural gap worth noting for the design doc: TransportBackend covers only the outbound half â€” inbound messages bypass the protocol via a messageHandler closure + TransportReceiver + receiveRootMessage, wired by convention at the composition root.

### 6. 2b. InspectorSession depends on the CONCRETE TransportSession actor, not a protocol; the protocol seam sits one level lower (TransportBackend)

`package protocol TransportBackend: Sendable { func sendJSONString(_:) async throws; func detach() async }` (Sources/WebInspectorTransport/TransportBackend.swift:3-6) is the only transport protocol. InspectorSession holds `let transport: TransportSession` in InspectorConnection (AttachedInspection.swift:239) and offers two package entries: `makeTransportSession(backend: any TransportBackend) -> TransportSession` (AttachedInspection.swift:501) and `connect(transport: TransportSession)` (AttachedInspection.swift:527). So the variation point for InspectorSession is 'any TransportBackend behind a shared TransportSession', i.e., TransportSession is the fixed routing/correlation engine and TransportBackend is the intended swap axis. Conformers today: NativeInspectorBackend (production, Sources/WebInspectorNativeTransport/NativeInspectorBackend.swift:7) and two test fakes (FakeTransportBackend, RecordingTransportBackend in DOMContainerTests.swift:1494).

Locations: Sources/WebInspectorTransport/TransportBackend.swift:3; Sources/WebInspectorCore/AttachedInspection.swift:501; Sources/WebInspectorCore/AttachedInspection.swift:527; Sources/WebInspectorNativeTransport/NativeInspectorBackend.swift:7

### 7. 2c. Inbound direction bypasses the TransportBackend protocol â€” the backend abstraction is half a duplex channel

TransportBackend declares only outbound (sendJSONString) and teardown (detach). Inbound delivery is wired by convention at the composition root: NativeInspectorBackend takes a `messageHandler: @Sendable (String) -> Void` in its init (NativeInspectorBackend.swift:17), the composition root points it at `TransportReceiver.receive` (InspectorSession+NativeAttachment.swift:23-25), and TransportReceiver (Sources/WebInspectorCoreSupport/TransportReceiver.swift:4 â€” note: lives in CoreSupport, not the Transport target) buffers/orders messages until `setTransport` then pumps them into `transport.receiveRootMessage(_:)` (TransportReceiver.swift:76, TransportSession.swift:98). A new backend implementer must reproduce this closure+receiver wiring manually; nothing in the protocol expresses it. Fatal-failure reporting is likewise a side-channel closure into `recordAttachmentError` (InspectorSession+NativeAttachment.swift:26-30).

Locations: Sources/WebInspectorNativeTransport/NativeInspectorBackend.swift:17; Sources/WebInspectorCoreSupport/TransportReceiver.swift:4; Sources/WebInspectorTransport/TransportSession.swift:98; Sources/WebInspectorNativeTransport/InspectorSession+NativeAttachment.swift:23

### 8. 3a. NativeBridge public C surface: fully public (C has no access control) and fully WKWebView-in-process specific

include/WebInspectorNativeBridge.h (only header, exported via publicHeadersPath: "include", Package.swift:115; also its own library product Package.swift:23-26) exposes: `WebInspectorNativeResolvedSymbols` â€” a struct of six raw uint64_t function addresses (connectFrontend, disconnectFrontend, stringFromUTF8, stringImplToNSString, destroyStringImpl, backendDispatcherDispatch); class `WebInspectorNativeBridge` with `initWithWebView:(WKWebView *)` designated initializer, `attachWithResolvedSymbols:error:`, `sendJSONString:error:`, `detach`, and messageHandler/fatalFailureHandler blocks; plus two ForTesting discovery functions. The implementation (WebInspectorNativeBridge.mm) reads WKWebView's obfuscated `_page` ivar (line 228-232), heap-scans the WebPageProxy for the InspectorController at cached offset 0x4C0 (lines 25, 259-404), and calls private WebKit symbols by raw address. The ABI is unusable for any non-WKWebView transport â€” its designated init REQUIRES a WKWebView.

### 10. 3c. What blocks an external remote-WebSocket transport today: access control at three layers, not architecture

(1) The seam protocol `TransportBackend` is `package` (TransportBackend.swift:3) â€” an external module cannot even name it, let alone conform. (2) Everything a backend must touch is `package`: TransportSession + init(backend:) (TransportSession.swift:3,23), receiveRootMessage (TransportSession.swift:98), ProtocolCommand/ProtocolEvent/ProtocolTarget.ID (TransportTypes.swift, ProtocolTypes.swift), TransportReceiver (CoreSupport). (3) The consumer entry is sealed: `InspectorSession` is `package final class` (AttachedInspection.swift:367), `connect(transport:)` is package (AttachedInspection.swift:527), and the ONLY public attach in the whole package is `WebInspectorSession.attach(to: WKWebView)` / `WebInspectorViewController.attach(to:)` (WebInspectorKit/WebInspectorNativeAttachment.swift:8,16), which hard-codes the native factory chain. Notably NOT blockers: the message envelope (plain JSON strings â€” shape-compatible with WebKit's remote inspector protocol), TransportSession's WebKit semantics (they are protocol-level, not native-bridge-level, see 1c), and the native C/symbol machinery (isolated behind NativeInspectorBackend; WebInspectorTransport has zero dependency on it).

Locations: Sources/WebInspectorTransport/TransportBackend.swift:3; Sources/WebInspectorCore/AttachedInspection.swift:367; Sources/WebInspectorCore/AttachedInspection.swift:527; Sources/WebInspectorKit/WebInspectorNativeAttachment.swift:8

### 11. 4. Verdict: transport is a real variation axis; the absorption point exists and is exercised (TransportBackend + TransportSession + InspectorSession.connect), but it is package-private and only half-expressed

Evidence the axis is real: two live implementations of TransportBackend today (NativeInspectorBackend in production, FakeTransportBackend/RecordingTransportBackend in tests) driving identical InspectorSession behavior; WebInspectorTransport compiles with Foundation only. The absorption point is the pair {`package protocol TransportBackend` (TransportBackend.swift:3-6) for outbound} + {`TransportSession.receiveRootMessage` (TransportSession.swift:98) via TransportReceiver for inbound} + {`InspectorSession.connect(transport:)` (AttachedInspection.swift:527) as the domain-layer plug}. For the rearchitecture: publishing the axis means (a) making TransportBackend + the ~8 envelope types (ProtocolCommand, .Result, ProtocolEvent, ProtocolDomain, ProtocolTarget.ID/Kind/Capabilities/Record, ProtocolFrame.ID, RuntimeContext.*, TransportSession.Error/Snapshot/MainPageTarget) public, (b) closing the inbound asymmetry â€” today a backend author must know the undocumented messageHandlerâ†’TransportReceiverâ†’receiveRootMessage convention copied at the sole composition root (InspectorSession+NativeAttachment.swift:13-34) â€” e.g., by folding inbound delivery into the backend contract, and (c) exposing a public connect on the session that accepts the abstraction, since the current public API only offers the WKWebView-native path.

Locations: Sources/WebInspectorTransport/TransportBackend.swift:3; Sources/WebInspectorTransport/TransportSession.swift:98; Sources/WebInspectorCore/AttachedInspection.swift:527; Sources/WebInspectorNativeTransport/InspectorSession+NativeAttachment.swift:13
